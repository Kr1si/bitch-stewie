## 1. What the project is

A LangGraph/deepagents orchestrator (FastAPI backend + React/MUI v7 frontend +
Postgres + Qdrant + draw.io, all Dockerized) that **delegates actual coding work
to Claude Code**, with CC repointed at **GLM 5.2 via Ollama** (no Anthropic key).
Target loop per the user: chat query → orchestrator plans → spawns CC to
implement → devises/runs tests → opens a merge request → **user reviews and
merges manually** → CI deploys → next query. Self-rewrite (the orchestrator
editing its own repo) is in scope. Cron-triggered workflows are a later
iteration. It is **not** fully autonomous — there is a human review gate before
push/merge.

Architecture decisions (locked, see memory `project-ai-assistant-architecture`):
GLM 5.2 via Ollama; Postgres not SQLite; never reimplement native CC features;
V1 = CC orchestration loop.

## 2. What's already built and working

The app is **live on krisiserver** (probed 2026-07-10, all containers healthy):

Implemented pieces of the loop:

  `/api/chat/resume/stream` (`backend/src/assistant/api/chat.py`).
- **Orchestrator** — `deepagents.create_deep_agent` with subagents
  architect / doc-writer / researcher / code-delegate
  (`backend/src/assistant/orchestrator/factory.py`). ReAct + todo list; no
  explicit plan node (fine for V1).
- **CC bridge** — `claude_agent_sdk` spawns CC at `cwd=repo_path`,
  `permission_mode="bypassPermissions"`, `setting_sources=["project"]`,
  `model=glm-5.2:cloud`, `ANTHROPIC_BASE_URL=http://ollama:11434`,
  `ANTHROPIC_AUTH_TOKEN=ollama` (`backend/src/assistant/cc_bridge/runner.py`).
  Post-impl it runs CC's native `/code-review` up to 3× and parses
  `{branch,commits,summary,tests}` from the result.
- **Pre-delegate approval gate** — `interrupt_on={"delegate_coding_task": True}`
  (`factory.py`) — the *only* HITL gate today.
- **Persistence** — Postgres checkpointer, Procrastinate jobs, `CCRun` /
  `CCRunEvent` / `Approval` / `Project` / `Decision` models
  (`backend/src/assistant/memory/models.py`).

## 5. Phase 1 plan (the code work that remains)

To close the loop:
1. **MR/PR creation** — new `open_merge_request` tool/step: after a delegation
   returns, push the branch and open a GitHub PR via `gh`/API. Add
   `GITHUB_TOKEN` + `ASSISTANT_GIT_REMOTE` to `config.py` and compose env.
2. **Post-implementation human review gate** — a second `interrupt_on` (or a
   `Command(resume=…)` step) after delegation returns, *before* push/PR. This is
   the user's review gate.
3. **Test gating** — V1: surface the delegation's `tests` result at the review
   gate and refuse to push on `failed`. A dedicated devise-test-plan / run-tests
   node is V1.5 (CC's skill already runs the suite).
4. **Self-rewrite targeting** — allow `delegate_coding_task` to target
   `/projects/bitch-stewie` with a stricter, always-on HITL gate for self-edits.
   Upgrades land only after user merge → CI rebuild → orchestrator restart
   (no hot-reload).

### Open decisions (defaults picked, confirm with user)
- **MR mechanism** → GitHub PR via `gh` + a `GITHUB_TOKEN` PAT (repo scope); user
  reviews/merges in GitHub; CI deploys on merge to main. **User must supply the
  PAT** (wire into compose env + GH Actions secret).
- **Self-rewrite scope** → confirm CC editing `/projects/bitch-stewie` on the
  server is in scope for V1, with an always-on human gate before any self-PR.
- **Test gating** → V1 blocks push on test failure; dedicated test node = later.

## 7. The plans/ convention

`plans/` (repo root) holds dated `YYYY-MM-DD-<slug>.md` files with frontmatter
`status: proposed|in_progress|done|dropped`, `created`, `project`. The
orchestrator writes them via `write_plan` and re-reads them via `list_plans` at
the start of an improvement session. Self-improvement plans target project
`bitch-stewie` and only reach the running orchestrator after merge → CI
redeploy (no hot-reload). See `plans/README.md` and the `plan-improvement` skill.
