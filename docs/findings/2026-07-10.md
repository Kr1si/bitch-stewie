# Findings — 2026-07-10

Handoff note: this captures the state of the **bitch-stewie** personal AI
assistant project as of 2026-07-10, so work can resume on a different machine.
It records what's built, what's live on krisiserver, the blockers, the **LLM
provider & DB setup (§3a)**, the loop gap analysis, what changed this session,
and the open Phase 1 plan + decisions. Read this first, then §3a, then
`docs/krisiserver.md` and `plans/README.md`.

## 1. What the project is

A LangGraph/deepagents orchestrator (FastAPI backend + React/MUI v7 frontend +
Postgres + Qdrant + draw.io, all Dockerized) that **delegates actual coding work
to Claude Code**, with CC pointed at **GLM via an Anthropic-compatible endpoint**
— Ollama by default, **Z.ai swappable via env with no code change** (no Anthropic
key). See §3a for the provider/DB setup.
Target loop per the user: chat query → orchestrator plans → spawns CC to
implement → devises/runs tests → opens a merge request → **user reviews and
merges manually** → CI deploys → next query. Self-rewrite (the orchestrator
editing its own repo) is in scope. Cron-triggered workflows are a later
iteration. It is **not** fully autonomous — there is a human review gate before
push/merge.

Architecture decisions (locked, see memory `project-ai-assistant-architecture`):
GLM 5.2 via Ollama; Postgres not SQLite; never reimplement native CC features;
V1 = CC orchestration loop.

## 2. What's already built and working

The app is **live on krisiserver** (probed 2026-07-10, all containers healthy):
ollama, backend (`127.0.0.1:8000`), frontend (`127.0.0.1:3000`), postgres,
qdrant, worker, drawio. Push-to-main → GitHub Actions → SSH deploy →
`docker compose up -d --build` works (backend rebuilt ~1h before probe).

Implemented pieces of the loop:
- **Chat** — SSE streaming at `POST /api/chat/stream` + resume at
  `/api/chat/resume/stream` (`backend/src/assistant/api/chat.py`).
- **Orchestrator** — `deepagents.create_deep_agent` with subagents
  architect / doc-writer / researcher / code-delegate
  (`backend/src/assistant/orchestrator/factory.py`). ReAct + todo list; no
  explicit plan node (fine for V1).
- **CC bridge** — `claude_agent_sdk` spawns CC at `cwd=repo_path`,
  `permission_mode="bypassPermissions"`, `setting_sources=["project"]`,
  `model=settings.cc_model`, and injects `ANTHROPIC_BASE_URL` /
  `ANTHROPIC_AUTH_TOKEN` / `API_TIMEOUT_MS` from settings
  (`backend/src/assistant/cc_bridge/runner.py`). Post-impl it runs CC's native
  `/code-review` up to 3× and parses `{branch,commits,summary,tests}` from the
  result. Provider is env-driven — see §3a.
- **Pre-delegate approval gate** — `interrupt_on={"delegate_coding_task": True}`
  (`factory.py`) — the *only* HITL gate today.
- **Persistence** — Postgres checkpointer, Procrastinate jobs, `CCRun` /
  `CCRunEvent` / `Approval` / `Project` / `Decision` models
  (`backend/src/assistant/memory/models.py`).

## 3a. LLM provider & DB setup (how it's configured)

### Two LLM consumers — know which is which

| Consumer | What it does | Config | Default backend |
|---|---|---|---|
| **Delegated CC** | the worker that writes code | `ASSISTANT_CC_*` env (§below) | Ollama, **Z.ai-swappable** ✅ |
| **Orchestrator planner** | the LangGraph deep-agent that plans/delegates | `ASSISTANT_DEFAULT_MODEL` (config.py default `ollama:glm-5.2:cloud`) | Ollama, **not yet repointed** |

Both must reach a working GLM endpoint for the loop to run.

### Claude Code provider (delegated worker) — env-driven, Ollama defaults

Configured in three layers (all overrideable, defaults preserve Ollama):
1. `docker/docker-compose.prod.yml` backend **and** worker `environment:` — the
   CC vars are `${ASSISTANT_CC_…:-ollama-default}`:
   ```yaml
   ASSISTANT_CC_ANTHROPIC_BASE_URL: ${ASSISTANT_CC_ANTHROPIC_BASE_URL:-http://ollama:11434}
   ASSISTANT_CC_ANTHROPIC_AUTH_TOKEN: ${ASSISTANT_CC_ANTHROPIC_AUTH_TOKEN:-ollama}
   ASSISTANT_CC_MODEL: ${ASSISTANT_CC_MODEL:-glm-5.2:cloud}
   ASSISTANT_CC_API_TIMEOUT_MS: ${ASSISTANT_CC_API_TIMEOUT_MS:-3000000}
   ```
2. `backend/src/assistant/config.py` — `cc_anthropic_base_url`,
   `cc_anthropic_auth_token` (default `"ollama"`), `cc_model`,
   `cc_api_timeout_ms` (default `"3000000"`). Env prefix `ASSISTANT_`.
3. `backend/src/assistant/cc_bridge/runner.py` — injects them into the CC
   subprocess env as `ANTHROPIC_BASE_URL` / `ANTHROPIC_AUTH_TOKEN` /
   `API_TIMEOUT_MS` (the auth token is no longer hardcoded).

**Protocol caveat:** CC speaks the Anthropic Messages API, so
`ANTHROPIC_BASE_URL` must be an Anthropic-compatible endpoint. Ollama and Z.ai
expose one; OpenRouter's default `/api/v1` is OpenAI-compatible and will NOT
work directly (would need a translating proxy like LiteLLM).

### To switch CC to Z.ai (the recommended path — no `ollama signin` needed)

On the server, create `/home/deploy/bitch-stewie/docker/.env` (gitignored —
survives `git reset --hard` across deploys). It is auto-loaded by compose when
run from the `docker/` dir:
```
ASSISTANT_CC_ANTHROPIC_BASE_URL=https://api.z.ai/api/anthropic
ASSISTANT_CC_ANTHROPIC_AUTH_TOKEN=<your Z.ai API key>
ASSISTANT_CC_MODEL=claude-sonnet-4-5
ASSISTANT_CC_API_TIMEOUT_MS=3000000
```
Z.ai auto-maps Claude model names → GLM (`claude-sonnet-4-5` → GLM-4.7), so use
a Claude name for `ASSISTANT_CC_MODEL` — do not pass `glm-5.2:cloud` to the Z.ai
endpoint. Then push to main → CI redeploys (or `docker compose up -d`). The key
never goes in the repo — only in the server's `docker/.env`.
Z.ai docs: https://docs.z.ai/scenario-example/develop-tools/claude

### How the DB / Qdrant / Ollama connections are configured

These are **NOT** driven by `docker/.env`. They are literal `environment:`
values in `docker-compose.prod.yml` (backend lines 59-61, worker 88-90),
pointing at the in-stack compose service hostnames:
```yaml
ASSISTANT_DATABASE_URL: postgresql://assistant:assistant@postgres:5432/assistant
ASSISTANT_QDRANT_URL: http://qdrant:6333
ASSISTANT_OLLAMA_BASE_URL: http://ollama:11434
```
Containers resolve `postgres` / `qdrant` / `ollama` on the compose network. To
change one (e.g. external Postgres), edit compose — or convert the line to
`${VAR:-default}` and promote it into `docker/.env`. `docker/.env.example`
documents all of this (overridable vs fixed-in-compose sections).

### Provider research: best way to serve GLM-5.2 to CC (2026-07-10)

Compared Z.ai direct vs Ollama cloud vs NVIDIA NIM for the same model GLM-5.2.
CC speaks the Anthropic Messages API, so protocol fit is decisive.

| | Z.ai direct `api.z.ai/api/anthropic` | Ollama cloud `glm-5.2:cloud` | NVIDIA NIM `integrate.api.nvidia.com/v1` |
|---|---|---|---|
| CC protocol | native Anthropic ✅ | native (Ollama Anthropic-compat) ✅ | OpenAI-only ❌ needs proxy |
| Hops | 1 | 2 (→Ollama→Z.ai) | 2+ (→proxy→NVIDIA) |
| Cost | $1.40/$4.40 per 1M or Coding Plan $18–160/mo | ~Z.ai (+ possible margin) | free 1000 credits, **not for prod** |
| Rate limit | Coding Plan quota or metered | via Ollama | **40 RPM** (→200) |
| Max output | ~128K | Z.ai out | 32K |
| Planner covered w/o code change? | no (repoint) | **yes** (already `ollama:glm-5.2:cloud`) | planner yes; CC needs proxy |
| Privacy | standard | via Z.ai | **free tier records data** ⚠️ |

**Three findings that decide it:**
1. **Z.ai's default CC mapping serves GLM-4.7, not GLM-5.2.** The official doc
   auto-maps sonnet/opus→GLM-4.7. To get GLM-5.2 + 1M context you MUST override
   the mapping, e.g. `ANTHROPIC_DEFAULT_SONNET_MODEL=glm-5.2[1m]`. `runner.py`
   does not set these envs yet — add them on the Z.ai path.
2. **Z.ai's Anthropic endpoint can drop nested tool-result blocks on long
   agentic loops** (known issue). Self-rewrite is exactly that. Z.ai's fallback
   is the OpenAI-compat endpoint, which CC can't speak. Watch-item.
3. **NVIDIA is OpenAI-only** → CC needs a translating proxy; free tier records
   data, 40 RPM, not-for-production. Great free sandbox, poor primary backend.

**Recommendation:**
- **Production:** Z.ai direct for CC + planner on Z.ai's OpenAI-compat endpoint
  (`ChatOpenAI` → `/api/paas/v4`) — one key, no Ollama, fewest hops, planner
  avoids finding #2. MUST add `ANTHROPIC_DEFAULT_SONNET_MODEL=glm-5.2[1m]` to
  CC env or you silently get GLM-4.7.
- **Fastest-to-working:** Ollama cloud — one `ollama signin` + pull covers CC
  AND the planner with zero code change (planner already defaults to it). Extra
  hop + possible margin. Good to get the loop live today.
- **Free validation:** NVIDIA NIM via `nvidia-claude-proxy` or LiteLLM — $0 to
  prove the loop end-to-end, then flip env to Z.ai. Don't run real self-rewrite
  on it (data recorded, 40 RPM).

**Suggested path:** start Ollama cloud (one command, loop live), migrate CC to
Z.ai direct once stable (add `glm-5.2[1m]` mapping env to `runner.py`).

### Cheaper planner model — stay on Z.ai (don't mix in DeepSeek)

The orchestrator **planner + sub-agents** do high-level planning/delegation, not
heavy codegen, so a lighter model is fine there. Z.ai has a ladder of lighter
coding/agent models incl. **free** ones — use one of those rather than adding a
second provider:

| Planner candidate | Price (in/out per 1M) | Context | Note |
|---|---|---|---|
| **GLM-4.7-Flash** | **FREE** | 200K | lightweight SOTA on SWE-Verified for its size; agent-oriented |
| GLM-4.5-Flash | FREE | 131K | strong reasoning/coding |
| GLM-4.5-Air | $0.20 / $1.10 | 128K | 12B active, tool-calling, CC-integrated |
| GLM-4.7 | $0.60 / $2.20 | 200K | SWE-Verified 73.8%, "think before acting" in CC |

All-Z.ai setup (preferred over a Z.ai+DeepSeek hybrid — one key, no proxy):
- **CC worker** → `glm-5.2[1m]` via Z.ai **Anthropic** endpoint (1M context,
  best long-horizon agentic coding).
- **Planner + sub-agents** → `glm-4.7-flash` (free) via Z.ai **OpenAI-compat**
  endpoint (`/api/paas/v4`, LangChain `ChatOpenAI`). 200K is plenty for planning.
- Fallback if the free tier rate-limits or reasoning is weak: GLM-4.5-Air
  ($0.20/$1.10) or GLM-4.7 ($0.60/$2.20) — same key.

DeepSeek V4 was considered as a cheap planner (V4-Flash $0.14/$0.28) but dropped:
Z.ai's free GLM-4.7-Flash beats it on cost and avoids a second key. DeepSeek V4
also trails GLM-5.2 on the long-horizon agentic benchmarks that matter for the
worker (FrontierSWE 29 vs 74, DeepSWE 8 vs 46), and its "3-5× cheaper" claim was
promo pricing that expired 2026-05-31 (V4-Pro list is $1.74/$3.48 — only ~20%
cheaper on output than GLM-5.2). NVIDIA NIM remains only a free sandbox
(OpenAI-only → needs proxy; 40 RPM; records data; not-for-production).

## 4. Loop gap analysis (target loop vs implemented)

| Stage | Status |
|---|---|
| chat → orchestrator plans | ✅ done (ReAct + todo) |
| spawn CC to implement | ✅ done |
| pre-delegate approval gate | ✅ done |

## 5. Phase 1 plan (the code work that remains)

To close the loop:
1. **MR/PR creation** — new `open_merge_request` tool/step: after a delegation
   returns, push the branch and open a GitHub PR via `gh`/API. Add
   `GITHUB_TOKEN` + `ASSISTANT_GIT_REMOTE` to `config.py` and compose env.
2. **Post-implementation human review gate** — a second `interrupt_on` (or a
   `Command(resume=…)` step) after delegation returns, *before* push/PR. This is
   the user's review gate.
3. **Test gating** — V1: surface the delegation's `tests` result at the review
   gate and refuse to push on `failed`. A dedicated devise-test-plan / run-tests
   node is V1.5 (CC's skill already runs the suite).
4. **Self-rewrite targeting** — allow `delegate_coding_task` to target
   `/projects/bitch-stewie` with a stricter, always-on HITL gate for self-edits.
   Upgrades land only after user merge → CI rebuild → orchestrator restart
   (no hot-reload).

### Open decisions (defaults picked, confirm with user)
- **MR mechanism** → GitHub PR via `gh` + a `GITHUB_TOKEN` PAT (repo scope); user
  reviews/merges in GitHub; CI deploys on merge to main. **User must supply the
  PAT** (wire into compose env + GH Actions secret).
- **Self-rewrite scope** → confirm CC editing `/projects/bitch-stewie` on the
  server is in scope for V1, with an always-on human gate before any self-PR.
- **Test gating** → V1 blocks push on test failure; dedicated test node = later.

## 7. The plans/ convention

`plans/` (repo root) holds dated `YYYY-MM-DD-<slug>.md` files with frontmatter
`status: proposed|in_progress|done|dropped`, `created`, `project`. The
orchestrator writes them via `write_plan` and re-reads them via `list_plans` at
the start of an improvement session. Self-improvement plans target project
`bitch-stewie` and only reach the running orchestrator after merge → CI
redeploy (no hot-reload). See `plans/README.md` and the `plan-improvement` skill.
